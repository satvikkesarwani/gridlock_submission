3. MODEL #1 — TrafficImpactModel (point regressor)  [model_pipeline.py]
--------------------------------------------------------------------------------
PURPOSE: predict incident CLEARANCE DURATION in minutes from event metadata.
  In the product UI this single number is surfaced as "Delay" (predicted_delay_mins).
  NOTE the naming nuance: model predicts clearance/duration time; the UI label calls
  it "Delay". [OPEN QUESTION] whether "delay" vs "clearance duration" is intentional.

ALGORITHM: single LightGBM regressor (LGBMRegressor) — n_estimators=300,
  learning_rate=0.05, max_depth=5, random_state=42, verbose=-1.
  Wrapped in sklearn Pipeline: ColumnTransformer(OneHot(cat)+StandardScaler(num))
  -> LGBMRegressor. Whole pipeline wrapped in TransformedTargetRegressor with
  func=np.log1p / inverse_func=np.expm1 (target log-transform; keeps preds positive,
  tames right-skewed tail).
  Docstring: plain LGBM beat a stacked XGB+LGBM+MLP ensemble in 5-fold CV
  (R^2 0.32 vs 0.23); stack overfits on ~2.8k rows, so kept simple.

FEATURES IN (13 total): see Section 2.
TARGET OUT: duration_mins (capped 480, log-space fit).

TRAINING ENTRYPOINT: `python model_pipeline.py` (__main__ block l.221-232).
  - Uses harmonized_master.csv if present, ELSE the Astram CSV.
  - train(): read csv -> preprocess_data -> X=FEATURES, y=duration clip 480 ->
    train_test_split 80/20 (seed 42) -> fit -> print MAE/RMSE/R^2/R^2(log) ->
    pickle.dump(self.model) to impact_model.pkl.
  - Reported metrics come from a single held-out 20% split (printed, not stored).
    evaluate.py provides the more honest 5-fold CV numbers.

ARTIFACT: backend/ml_components/impact_model.pkl (387 KB). The ENTIRE fitted
  TransformedTargetRegressor (preprocessing + model) is pickled, so loading needs
  the same sklearn/lightgbm versions [compat risk].

RUNTIME LOAD: main.py:26-29 constructs TrafficImpactModel() and calls .load();
  warns if missing. load() pickle.loads impact_model.pkl, sets is_fitted=True.

RUNTIME PREDICT: TrafficImpactModel.predict(event_cause, priority, hour_of_day,
  is_weekend, requires_road_closure, event_type='unplanned',
  corridor='Non-corridor', veh_type='unknown', month=6):
  - Builds a synthetic single-row DataFrame with a fake timestamp
    (year=2024, month, day=15, hour=hour_of_day, UTC).
  - Runs _engineer_features to make cyclical/time features, then OVERRIDES
    is_weekend with the caller's value (l.161-162). day_of_week/is_rush_hour still
    come from the synthetic date (day=15), so day_of_week is whatever weekday the
    15th of `month` falls on — NOT necessarily consistent with is_weekend.
    [OPEN QUESTION / minor inconsistency worth noting in report.]
  - Predicts, returns round(float(pred),1). Cast to python float => JSON-safe.
  NOTE: the API only passes the first 5 args; event_type/corridor/veh_type/month
  use DEFAULTS for every production prediction. So 3 of the 5 categorical features
  are effectively constant at inference time (event_type='unplanned',
  corridor='Non-corridor', veh_type='unknown'). Only event_cause + priority +
  hour + is_weekend + road_closure actually vary the prediction.

4. MODEL #2 — ClearanceRangeModel (quantile regression)  [quantile_model.py]
--------------------------------------------------------------------------------
PURPOSE: clearance-time RANGE, not a point — optimistic (P10), expected (P50),
  pessimistic (P90) minutes. For operator triage / confidence.
WHY QUANTILE over survival: docstring l.9-14 — survival needs a trustworthy
  "still-open at T" timestamp; the only candidate (modified_datetime) is a
  record-edit time (median 5.7 min after start), so survival is miscalibrated.
  Quantile regression needs only closed incidents (which exist).

ALGORITHM: 3 independent LGBMRegressor(objective='quantile', alpha in {0.10,0.50,
  0.90}), same n_estimators=300/lr=0.05/depth=5/seed=42, each in its own
  ColumnTransformer pipeline (same FEATURES as model #1). NO log-transform here
  (trained directly on duration in minutes, clipped at 480).

CALIBRATION: docstring states P10-P90 SHOULD cover ~80% of incidents; MEASURED
  coverage is ~73% (so slightly under-calibrated; honest disclosure in code).

TRAIN: `python quantile_model.py` -> reuses TrafficImpactModel().preprocess_data,
  fits 3 models, pickle.dump dict{name->pipeline} to quantile_model.pkl (1.2 MB).
  Trains on the Astram CSV (DATA constant), NOT harmonized_master.

PREDICT: same input signature as model #1. Builds same synthetic row, predicts all
  3 quantiles, then np.sort to ENFORCE monotonicity p10<=p50<=p90 (because the 3
  models are independent and could cross). Returns dict:
    expected_clearance_mins  = round(p50,1)
    optimistic_p10_mins      = round(max(p10,0),1)   (floored at 0)
    pessimistic_p90_mins     = round(p90,1)
  All cast to python float => JSON-safe. If not trained/loadable -> raises
  RuntimeError (caught by API -> HTTP 500).

5. ResourceOptimizer (HEURISTIC — NOT machine learning)  [model_pipeline.py:168]
--------------------------------------------------------------------------------
- Pure rule-based static function recommend(event_cause, priority,
  requires_road_closure, attendance_str).
- Base: police=2, volunteers=0, barricades=2, diversions=0. Adds fixed bumps:
    public_event / political_rally: +10 police, +15 vol, +20 barr, +2 div
    accident: +4 police, +5 barr, +1 div
    vehicle_breakdown: +1 police, +2 barr
    priority High: +5 police, +10 barr
    requires_road_closure: +4 police,+5 vol,+15 barr,+2 div
    attendance: police += att//5000, volunteers += att//2000 (parse '45,000'),
      bare except swallows parse errors.
- Returns sworn_staff, volunteers, barricades, diversions, relief_factor
  (=round(1.2+police*0.01,2)), estimated_budget (=police*50+vol*20+barr*10).
- [MISMATCH] Frontend eventTypes.js maps UI labels to causes: public_event,
  protest, procession, vip_movement. ResourceOptimizer only special-cases
  public_event / political_rally / accident / vehicle_breakdown. So 'protest',
  'procession', 'vip_movement' get ONLY the base + priority/closure bumps — no
  cause-specific resources. Worth flagging as a contract gap.

9. KEY RISKS / LIMITATIONS (for the report's honesty section)
--------------------------------------------------------------------------------
- Small training set (~2.7k usable rows); model R^2 ~0.32 raw (modest). Point
  predictions have ~54 min MAE per the code comment. Not production-accurate.
- ~66% of incidents are right-censored (no end time) and DROPPED -> selection bias
  toward incidents that were actually closed/recorded.
- 3 of 5 categorical features (event_type, corridor, veh_type) are CONSTANT at
  inference (API never passes them) -> model can't use that learned signal live.
- Frontend only varies event_cause + attendance; priority/hour/closure are fixed
  defaults -> live predictions vary less than the model technically allows.
- Quantile band under-calibrated (~73% vs target 80%).
- Synthetic timestamp at predict time; day_of_week/is_rush_hour derived from a fake
  date (day 15 of month) while is_weekend is overridden separately -> internal
  feature inconsistency.
- harmonize_data.py fabricates clearance times (rule + random noise) -> if used,
  trains on synthetic labels. Default path does NOT use it (file absent).
- /api/debrief and /api/live-status are MOCK (random/hardcoded), incl. a fake
  "Model R2: 0.89" — must not be presented as real model performance.
- encoders.pkl appears unused (needs grep confirmation).
- Pickle artifacts tie deployment to exact sklearn 1.7.1 / lightgbm 4.6.0 versions.
- CORS allow_origins=["*"] (fine for demo, not for prod).
- Errors collapse to HTTP 500 with raw str(e) (info leak; no structured errors).
